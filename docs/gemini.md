# Gemini 评估报告：Cheng 硬实时（HRT）后端

## 1. 执行摘要

Cheng 硬实时（HRT）后端设计（`cheng-hard-realtime-backend.md`）通过“做减法”（禁用动态特性）而非 Rust 的“做加法”（复杂的借用检查），在特定的安全维度上展现了达到甚至超越 Rust 的潜力。它旨在构建一个类似 **SPARK Ada** 或 **MISRA C** 的可验证子集。

然而，通过与 `cheng-formal-spec.md` 的交叉评估，我们发现设计在原子操作、并发模型和异常处理策略上存在核心语义冲突，必须解决这些问题才能确保在目标硬件上物理可实现。

---

## 2. 安全性分析 vs. Rust

| 特性 | Rust (Safe Rust) | Cheng HRT (设计目标) | 结论 |
| :--- | :--- | :--- | :--- |
| **堆内存管理** | **安全且动态**。使用 `Box`/`Rc`/`Arc` 配合生命周期检查。 | **严格静态**。“禁止隐式堆分配”，“仅限固定容量”。 | Cheng HRT 更严格。它通过牺牲灵活性，彻底消除了 OOM（内存耗尽）和碎片化风险。 |
| **时间保障** | **尽力而为**。无 GC，但无内建 WCET 保证。 | **有保证**。显式的 `@wcet`、`@bound` 循环、无 GC、无隐式分配。 | 在 *时间* 安全性上，Cheng HRT 优于 Rust（前提是硬件配合）。 |
| **数据并发安全** | **类型系统 (Send/Sync)**。编译期防止数据竞争。 | **静态任务**。目前对全局状态缺少显式的数据竞争防护。 | **存在缺口**。如果没有对 `@task` 共享状态强制 `Sync` 检查，HRT 在这方面不如 Rust 安全。 |

---

## 3. 正式规范 (Formal Spec) 对齐与冲突

对照 `cheng-formal-spec.md` (v2026-01-22)，我们发现了四个关键的设计冲突：

### A. 原子性陷阱 (ORC vs. WCET)
*   **规范**：`CHENG_MM_ATOMIC=default` （默认使用原子引数）。
*   **冲突**：在现代 MMU/多核架构上，原子 RMW（读-改-写）操作会引入显著且不可预测的总线锁定开销，破坏 WCET 的可预测性。
*   **修正**：HRT 模式必须强制设定 `CHENG_MM_ATOMIC=0`。

### B. Async 语义断层
*   **规范**：`async`/`await` 是核心语言特性，依赖特定的运行时模型。
*   **冲突**：HRT 定义了静态 `@task` 模型，但未定义如何处理 `async` 代码。
*   **修正**：在 HRT Profile v1 中明确禁用 `async`/`await` 关键字，或在未来将其映射为静态状态机。

### C. 并发安全缺口
*   **规范**：定义了 `Send`/`Sync` trait 用于保障线程安全。
*   **冲突**：HRT 禁止动态线程（`spawn`）但允许静态 `@task`。规范目前未强制对任务间共享的数据进行 `Sync` 检查，留下数据竞争隐患。
*   **修正**：编译器必须将 `@task` 的上下文切换视为并发边界，强制要求所有全局共享变量满足 `Sync` 约束。

### D. Panic 策略
*   **规范**：`?` 操作符在 `Err` 时会触发立即返回或 **panic**。
*   **冲突**：在嵌入式/硬实时场景中，Panic 通常意味着系统死机或看门狗复位，是不可接受的。
*   **修正**：HRT 模式必须静态拒绝可能导致 panic 的 `?` 操作，或强制使用能保证无 panic 路径的 `Result` 处理方式。

---

## 4. 关键风险

1.  **验证的深渊**：依赖“静态验证 ARC”在数学上极具挑战性。若没有完整的借用检查器，极易导致误报（不可用）或漏报（不安全）。
2.  **硬件不可测性**：在 AArch64（Android/iOS）等目标平台上，多级缓存和乱序执行使得“硬实时”更多是一种概率保证而非确定性保证，无论软件栈如何设计。
3.  **生态割裂**：采用“严格子集”策略会导致大量标准库（依赖动态 `seq`、字符串拼接等）不可用，面临生态分裂风险。

---

## 5. 战略建议

### I. 采用区域内存模型 (Arena)
与其死磕静态验证 ARC，不如将 **Arena** 引入为 HRT 的一等公民。
*   **机制**：每个 `@task` 拥有预分配的线性缓冲区。分配复杂度 O(1)。任务周期结束时统一切换/复位。
*   **收益**：无需 ARC 验证，消除碎片，保证 WCET。

### II. 利用效应系统 (Effect System)
不要仅依赖全局的 `--mode:hrt` 开关。利用编译器追踪副作用：
*   例如：`fn foo() @no_alloc @no_block { ... }`
*   这将允许那些*恰好*符合安全的现有标准库函数自动被复用，减少生态割裂。

### III. 强制任务间通信的 `Sync` 约束
对全局变量套用 Rust 的 `Mutex`/`RefCell` 模式。
*   **规则**：HRT 任务中禁止直接访问可变全局变量。
*   **方案**：必须使用实现了 `Sync` 的安全包装器（如 `Atomic[T]`, `IrqSafeCell[T]`）。

### IV. 定义目标层级 (Target Tiers)
根据硬件能力明确区分保证等级：
*   **Tier 1 (MCU/R-Core)**：硬实时 (保证 WCET)。
*   **Tier 2 (A-Core/x86)**：确定性延迟 (无 GC/内存分配尖峰)，但受限于硬件复杂性，不承诺严格微秒级 WCET。
