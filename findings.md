# 当前发现

- 当前调用链上的真根已经切到两类：
  - unresolved known call 之前没有统一 reason，`v3ParserAppendCallExprsLine(...)` 会把“命中名字但没解到 target”的情况也写成 `resolved=1`。
  - direct import 的 external call 名之前只收 qualified 名，不收 unqualified 名，所以 ambiguous external call 会直接漏出 `external_call` 入口。
- 这轮又钉住了两个更细的真根：
  - alias import 之前也会错误贡献 unqualified external 名，导致 unqualified resolve 被 alias 污染。
  - `maxQualifiedCallDepth` 之前会先于 direct unqualified 名收集生效，导致没有 dotted call 的 source 根本收不到 direct import external 名。
- 这轮再补上的真根：
  - qualified target 的 qualifier 归属之前只取第一条 import edge；只要两个 direct import 推导出同一个 qualifier，就会静默选第一条，而不是显式报 `ambiguous_qualified_target`。
- 这轮继续补上的真根：
  - aliasless direct import 的 unqualified external 名之前不会避开当前 source 的同名本地函数，所以同一行 `SharedEcho(...)` 会同时被记成 `local_call` 和 `external_call`。
- 这轮再补上的真根：
  - qualified external name 即使已知，也不能绕过当前函数里的参数和局部绑定；像 `typednested.plainhelper.TypedExternalLen(...)` 这类调用，如果 `typednested` 已经是局部名，就必须退回 `member_call`，不能继续按 import qualifier 解析。
- 这轮还钉住了一个测试面真根：
  - `compiler_csg_smoke / lowering_plan_smoke` 直接 import lowering rule 枚举并自己解释，会把 ordinary compile 闭包拉胖到超时；这类 smoke 该直接读 report 真源，而不是在测试里重建第二份规则解释。
- 另外一个容易误用的点也已经钉住：`v3ParserReadNormalizedExprLayerFromPath(...)` 只走本地文本层，不带 import edge；要验 external/qualified call，必须走 `v3ParseOrdinarySourceStub(...)`。
- 现在 resolved/unresolved 两侧都有明确 gate 了，后面再露红会直接指向“没 target”还是“没 reason”。
- 这轮又钉住了一个 parser 真根：
  - `v3ParserCollectExternalCallNames(...)` 和 `v3ParserResolveUnqualifiedExternalCallTarget(...)` 如果只修一边，closure-visible unqualified external call 还是会一半可见、一半不可解；名字收集和 target resolve 必须一起沿同一条 aliasless direct import closure 走。
- 这轮还钉住了另一处更隐蔽的真根：
  - `v3ParserReadNormalizedExprLayerFromTextWithKnownCallsAndProfiles(...)` 只要还留着旧的 `local/external/importc/member` 分裂扫描，就会把已经修好的 closure-visible 归因重新绕开，而且旧路径里的短路索引判断会在 import 行上直接炸 `idx=len`。
- 这轮再补上的一处真根：
  - `ResolveQualifiedCallTarget(...)` 之前只看 target profile 的直接 local/importc 声明，不看 target 自己 aliasless direct-import closure，所以 `mid.Foo(...)` 这类 qualified target 会比 unqualified target 少一整层可见函数。
- 与它配套的名字面真根也一起补了：
  - `CollectExternalCallNamesRecursive(...)` 之前只会为 qualified target 追加 `mid.direct_decl(...)` 和 `mid.leaf.Func(...)`，不会补 `mid` 自己 closure-visible 的 unqualified 名，所以 parser 连 `mid.Foo(...)` 这层入口都识别不到。
- 这轮又钉住了一个 lowering 真根：
  - `typed lowering rule` 之前只有计数和结构体，没有稳定文本 key；`compiler_csg` 和 `lowering_plan` 虽然都能算规则，但 report 面上看不到“到底是哪条规则”，后面 seed 想真接线时还是会逼着消费侧自己再解释一遍。
- 这轮继续钉住的 seed 真根：
  - wasm call 这条高层分发之前虽然已经有 `arg slot / result kind / import kind` helper，但 `register_import / type_index / analyze / emit` 还是各自现算一遍。只要其中一处晚一步改，wasm caller-side 语义就会重新分叉。
- 这轮再补上的 caller-side 真根：
  - native/importc 这条链之前虽然已经有 `arg kind / return kind / result kind` helper，但 `prepare/scalar call/call-into-address/call-from-spills/ffi_handle return fixup` 还是各自现场重算一遍。只要其中一处晚一步改，caller-side 语义就会重新分叉。
- 这轮继续补上的 caller-side 真根：
  - composite return caller 入口之前还是“resolve target/rule 成功后，只看 `result_uses_address` 这个单布尔值”再各处各自决定怎么落到目标地址。这样普通 composite return 和 import-buffer 式 composite return 在更高层还是没有单独规则名，后面一旦有人只修其中一条，很快又会分叉。
